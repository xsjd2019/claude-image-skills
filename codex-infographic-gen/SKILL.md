# Codex Infographic Generation

会話の文脈で出てきた情報を **人間にとって理解しやすい1枚のインフォグラフィック** にしてダウンロードフォルダに置く。**情報量を惜しまず**、コンテンツに合った**複数のビジュアル技法を組み合わせる**のがこのスキルの目的。

## いつ使うか — codex-design-to-code との使い分け

| | codex-infographic-gen（このスキル） | codex-design-to-code |
|---|---|---|
| 用途 | 会話の内容を1枚にまとめる | LP / デザインカンプ / UI素材 |
| 出力 | 1枚のWebP画像のみ | PNG群 + HTML/CSS実装 |
| 並列生成 | しない | する |
| 保存先 | `~/Downloads/` | プロジェクト or `~/Downloads/` |

会話の途中で「これ図にして」「インフォグラフィックにまとめて」と言われたら基本このスキル。LP作成や複数アセットが必要なら codex-design-to-code へ。両者は混在させない。

## 4つのインフォグラフィックタイプ（あくまでヒント）

タイプは「主たる構造」のヒント。**1枚の中で複数の技法を組み合わせて構わない**（むしろ推奨）。迷ったら `summary` を選ぶ。

| タイプ | 向いている内容 | 中心となる構造 |
|---|---|---|
| `concept` | 概念・仕組み・関係性の説明 | 関係図 / レイヤー図 / 仕組み解説 |
| `summary` | 議論や検討のまとめ | 結論 + 根拠 + 数字 + 次アクション |
| `comparison` | 選択肢・トレードオフ・対比 | 比較表 + メリデメ並列 |
| `flow` | 手順・プロセス | ステップフロー + 各ステップ詳細 |

## 良いインフォグラフィックの原則（最重要）

質の高い和文インフォグラフィックは、以下の特徴を持つ。**このスキルではこれらを実現することが目的**。テンプレ的な「番号カード並列」に逃げない:

1. **情報量を惜しまない** — 1枚で「全部わかる」と読み手が感じる密度。数字・固有名詞・具体例・比較を入れる
2. **セクションごとに違う技法を使う** — 同じ形のカードを5つ並べるのではなく、ある部分は表、別の部分はフロー、別の部分はコールアウト、というように変える
3. **結論や重要な数字をハイライト** — 「¥0」「年間120万円」のように重要な数字や結論を装飾枠で目立たせる
4. **具体例を必ず入れる** — 抽象論だけにせず「Aさんの場合」のようなシナリオボックスを設ける
5. **比較すべきは比較表で** — ○×、数値、項目別の対比は表で
6. **手順は矢印付きフローで** — 番号 + 矢印 + 各ステップの説明
7. **人間が登場する内容ならキャラクター図を使う** — 会社員・社長・顧客などをアイコン化
8. **タイトルは主張型に** — 「節税の仕組み」より「給与の一部を日当に置き換えるだけで会社も個人も税負担を軽減」のような主張型

### 使える表現技法（codex に渡す候補リスト）

- **比較表**（○× / 数値比較 / 列対比）
- **ステップフロー**(矢印付き、横/縦)
- **並列カード**(2-3列のメリデメ対比など)
- **ハイライト/コールアウト**(重要数字・結論を装飾枠で)
- **具体例ボックス**(「Aさんの場合」「シナリオX」のような事例)
- **キャラクターイラスト**(会社員・社長・顧客のシルエット)
- **関係図**(エンティティ間を線・矢印で結ぶ)
- **タイムライン**
- **チェックリスト**
- **アイコン付きラベル**
- **注釈の小箱**(※補足、💡ポイント、⚠注意 など)

### アンチパターン（やってはいけない）

- 同じ形・同じサイズの番号付きカードを5つ縦に並べただけ
- 各カードに「見出し1個 + 1-2行の説明 + 装飾アイコン1個」のワンパターン
- 数字も具体例もない、抽象的な見出しと一般論だけ
- タイトルが「〜について」「〜まとめ」のような無味乾燥なトピック名

## トンマナ（プリセット制）

インフォグラフィックの配色・タイポグラフィ・装飾感は、`references/styles/<name>.md` に定義された**プリセット**で統一する。これによりブランド一貫性を確保しつつ、用途別に切り替えられる。

- **デフォルト**: `references/styles/default.md` — 明るいプレゼン資料調（白背景・パステルカード・カラフルアクセント・人物イラスト）
- **ユーザー指定**: 「シニア向けで」「コンサル調で」のような指定があれば、対応する `references/styles/<name>.md` を読む（存在しない場合は default にフォールバック + ユーザー通知）
- **新規プリセット追加**: `references/styles/<name>.md` を追加するだけで使えるようになる

プリセットファイルには**人間が編集時に参照するためのサンプル画像**（`<name>-sample.png` など）が同梱されることがあるが、**Claude や codex はこれを Read しない**。コンテキストを膨らませないため、トンマナはプリセット `.md` のテキストのみで codex に伝える。

## ワークフロー

```
1. 抽出: 会話から「読み手に伝えるべき情報」をリッチに集める
   → 内容量を見て、1枚で収まるか複数枚に分けるかを判断（基本は1枚）
2. 宣言: 何を/どんなタイプで作るか、何枚で作るかを1行で報告
3. トンマナ読み込み: references/styles/default.md を Read（ユーザー指定があればそのプリセット）
4. プロンプト作成: /tmp/codex-prompt-<slug>.txt に {{PNG_PATH}} プレースホルダー付きで書き出す
   → 複数枚の場合は各ページ分のプロンプトを別ファイルで作成
5. 実行: bash scripts/generate.sh <slug> /tmp/codex-prompt-<slug>.txt を run_in_background で起動
   → 複数枚の場合は同じ呼び出しを並列で N 回
6. 報告: 完了通知を受け取り、Read で確認後、全ページのパスとサイズをユーザーに返す
```

**出力サイズは 1024x1536（縦長、A4 portrait に近い 1:1.5 比率）**。これは gpt-image-2 の portrait ネイティブサイズ。厳密な A4 比率（1:1.4142）にトリミングする方式も試したが、codex が「下端は余白」というピクセル境界を厳守できず content loss が起きるため**クロップは廃止**した。1024x1536 のまま出力し、縦長ドキュメントとして使う。

### Step 1 — 抽出（リッチに、構造化せずに）

会話の直近のやりとりを読み、読み手に伝えるべき情報を**惜しまず集める**:

- 主要な主張・結論
- それを支える具体的な数字・固有名詞・日付・金額
- 比較対象があれば、その対比軸と各値
- 手順があれば、各ステップの名前と内容
- メリット/デメリット、リスク、注意点
- 具体例・事例（「Aさんの場合」のような）
- 補足や注釈

**重要な禁止事項**:

- ❌ 「ステップ1の見出し: ○○、本文: △△」のように、Claude 側で**構造を作り込んで codex に渡さない**
- ❌ 「縦長5ステップで描く」「番号を大きな丸で」「見出しは太字、本文は短く」のような**レイアウト指示を Claude 側で書かない**
- ❌ 配色を細かく指定しない（トンマナはプリセットで既に伝わる）

これらをやると codex は単なる文字流し込みエンジンになり、テンプレ化した貧弱な図しか出てこない。**生情報・コンテキスト・主張・数字を渡すだけ**にして、構造化・レイアウト判断は codex に完全委任する。

**要素数の上限も設けない**。情報量が多くても、codex 側が適切な視覚化でレイアウトする。ただし本筋から逸れる枝葉は削ぎ落とす。

**情報密度 vs 可読性のバランス**: 1ページ（1024x1536）にセクションを 8 個以上詰め込むと、各セクションの文字が小さくなって読みにくくなる傾向がある。**1ページの目安は 5〜7 セクション**。

### 複数枚に分けるかの判断（基本1枚、必要に応じてN枚）

**1枚で収まる場合**（デフォルト）:
- セクション数 5〜7 個以内
- 各セクションの内容が短く、1ページ（1024x1536）に余裕で収まる
- 視覚的に「詰めても読める」密度

**複数枚に分けるべきサイン**:
- ❌ セクション数が 8 個以上、削れない
- ❌ 各セクションの内容（数字・比較表・具体例）が多くて、1ページに圧縮するとフォントが小さくなりすぎる
- ❌ 余白が極端に狭くなる（読み手の目が休まらない）
- ❌ 論理的に独立した2つ以上のトピックを1枚に押し込もうとしている

**分割の仕方**:
- 論理的なまとまりで分割（例: P1=概要・前提、P2=詳細比較、P3=結論・アクション）
- 各ページが「単体でも読める」自己完結性を持たせる
- ページ番号（P1/3, P2/3...）を各ページのタイトル付近に入れるよう codex に指示
- slug は `<topic>-p1of3`, `<topic>-p2of3`, `<topic>-p3of3` のように番号を含める
- **コンテンツ量をページ間でなるべく均等に配分**する。1ページがぎっしりでもう1ページがスカスカ、は禁止

**ページ間の整合性（重要）**:
複数ページのときは、**各ページのプロンプトに以下を必ず追加**する:

```
# ページ整合性
- これは N 枚組のうち P 枚目
- 全ページで同じ紙面密度・余白・タイポグラフィスケールを使う
- レイアウト構成も揃える（左右マージン、見出しサイズ、カードのpadding 等）
- ページごとに「ぽつんと中央寄せ」「ぎっしり詰め込み」のばらつきが出ないように
```

これがないと、コンテンツが少ないページは codex が勝手に中央寄せで余白多めに描いて、ページごとに見た目がバラつく（実際にあった失敗）。

**実装**: Claude が複数の `bash scripts/generate.sh` 呼び出しを **並列で** run_in_background する。各ページが独立したプロンプトで、それぞれの task-notification を受け取る。

ユーザーが対象を明示している（例:「A案とB案を比較して」）ならそれを尊重。指定がなければ Claude が会話の中心テーマを判断する。

### Step 2 — 宣言

1行で何を作るかを宣言:

```
「{タイトル}」を {タイプ} 図でまとめます。生成開始します。
```

ユーザーが「いやそうじゃない」と即軌道修正できる程度の情報を出す。

### Step 3 — トンマナ読み込み

`references/styles/default.md` を Read して、その**全文**を Step 4 のプロンプトの「# トンマナ」セクションに埋め込む。

ユーザーが特定のプリセットを指定した場合（「シニア向けで」「テック調で」など）は、対応する `references/styles/<name>.md` を読む。存在しなければ default にフォールバックし、その旨をユーザーに伝える。

### Step 4 — プロンプトテンプレート作成

`/tmp/codex-prompt-<slug>.txt` に Write でプロンプトを書き出す。**重要**: PNG 保存先は `{{PNG_PATH}}` プレースホルダーで指定する（scripts/generate.sh が実パスに置換する）。slug は内容を表す短い英語（kebab-case、例: `auth-flow`, `tax-saving-scheme`）。

#### プロンプトテンプレート

```
高品質な日本語インフォグラフィック画像を生成し、{{PNG_PATH}} に保存してください。
組み込みの画像生成ツール（image_gen）を使ってください。

# 背景・目的
<このインフォグラフィックが必要な背景。誰がどんな状況で見るか、何のために作るか>

# 主たる主張・読み手に得てほしい結論
<1〜3行で。これを軸にあなた（codex）がタイトル・上部キーメッセージを設計してください>

# 情報の素材（生情報。あなたが必要なものを選び、適切な視覚技法に落とす）

【背景・コンテキスト】
<議論の経緯、対象、前提など。長くてOK>

【数字・固有名詞・日付・金額】
<具体的な値をすべて列挙>

【比較対象（あれば）】
<選択肢A、B、C / プラン1、2 などの差分>

【手順・プロセス（あれば）】
<順序のあるステップ>

【具体例・シナリオ】
<「Aさんの場合」「ケースX」など、必ず1つは入れたい>

【メリット / デメリット / リスク / 注意点】
<>

【特に強調したい点】
<コールアウトで目立たせたい数字や結論>

【補足・注釈】
<※や💡で添える情報>

# 視覚化はあなた（codex）が判断する
- レイアウト、配色、装飾、どのセクションをどう見せるかは、上の情報を読んで最適な方法を選んでください
- 「同じ形・同じサイズのカードを縦に並べるだけ」の単調なレイアウトは避ける
- セクションごとに、内容に最も合った視覚技法を使い分ける
  （比較表 / ステップフロー / コールアウト / 並列カード / 関係図 / タイムライン /
   キャラクター図 / 具体例ボックス / チェックリスト / 補足の小箱 など、適切なものを自由に組み合わせる）
- 重要な数字・結論は装飾的に目立たせる
- 人間が登場する内容ならキャラクターアイコン（会社員・顧客・社長などのシルエット）を活用する

# 守ってほしい制約
- サイズ: 1024x1536（縦長ドキュメント、A4 portrait に近い 1:1.5 比率）
- **紙面の全体を埋める**:
  - 左右マージンは 30〜50px 程度に抑える（余白を狭く）
  - 上下マージンも 50〜80px 程度
  - 紙面の上端から下端まで、コンテンツで満たす（**重要な要素を下端ギリギリに置いても OK、クロップはされない**）
  - 各セクション・カード・ボックスは大きめに、たっぷり配置して紙面全体を満たす
  - 「ぽつんと中央寄せ」「広い空白」は禁止。コンテンツが少なくても、各要素を大きく/カードに余白を持たせる形で紙面を埋める
  - フォントサイズは紙面サイズに対して**十分大きく**（タイトル 60〜80pt 相当、見出し 28〜36pt 相当、本文 14〜18pt 相当の見え方）
- 日本語で書く（明朝NG）
- 下の「トンマナ」セクションに従って配色・タイポ・装飾を統一する

# トンマナ（このスタイルに合わせて生成）
<ここに references/styles/<preset>.md の全文を貼る>

完了したら絶対パスと「===CODEX_DONE===」を出力してください。
```

### Step 5 — generate.sh 実行（パイプライン一発）

scripts/generate.sh が codex 起動 → PNG 待ち（タイムアウト + サイズ検証）→ WebP 変換 → 無音通知 → 結果出力 までを一気通貫で実行する。

**単一ページの場合**:
```bash
bash ~/.claude/skills/codex-infographic-gen/scripts/generate.sh \
  <slug> /tmp/codex-prompt-<slug>.txt
```

**複数ページの場合** — 各ページを並列起動（Claude が同じ応答で複数の Bash 呼び出しを発行）:
```bash
# Bash call 1 (run_in_background: true)
bash scripts/generate.sh <slug>-p1of3 /tmp/codex-prompt-<slug>-p1.txt

# Bash call 2 (run_in_background: true, 同じ応答で発行)
bash scripts/generate.sh <slug>-p2of3 /tmp/codex-prompt-<slug>-p2.txt

# Bash call 3 (run_in_background: true)
bash scripts/generate.sh <slug>-p3of3 /tmp/codex-prompt-<slug>-p3.txt
```

各ページは独立した codex プロセスで並列生成される。完了は別々の task-notification で来るので、全部受け取ってから Read してまとめて報告。

成功時の stdout 最終行:
```
✅ ~/Downloads/infographic-<slug>-<timestamp>.webp (XXX K)
```

### Step 6 — 結果確認と報告

task-notification を受け取ったら:
1. stdout の最終行から WebP パスを取得
2. `Read` でファイルを開いて視覚的に確認（文字崩れ・致命的な誤字がないか）
3. ユーザーに報告:
```
✅ ~/Downloads/infographic-<slug>-<YYYYMMDD-HHmm>.webp (XXX KB)
```

崩れや小さい文字が見つかったら、ユーザーに通知して再生成案を提示。

### プロンプトの良い例 vs ダメな例

**ダメな例 ①（構造を作り込みすぎ。codex が「文字流し込みエンジン」になる）:**

```
タイトル: arb 商談デモ フロー
サブタイトル: 1時間の対面商談で使う適応型2モード

縦長5ステップのフロー図として描く:
  ステップ1 (見出し『ヒアリング』 / 本文『AIツール利用の有無を確認』)
  ステップ2 (見出し『スコア診断』 / 本文『顧客の本物のHPを点数化』)
  ...

スタイル:
  - 配色は紺・オレンジ・白系
  - ステップ番号を大きく丸で配置、見出しは太字、本文は短く
```

→ 結果: 言われた通り「丸で番号 + 太字見出し + 1行本文」のカードが5つ縦に並ぶ単調な図。

**良い例（コンテキストを豊富に、視覚化は codex に委ねる）:**

```
背景・目的: SaaS製品『arb』の新規商談用デモシステムの全体像を、
営業チーム新人オンボーディング資料として1枚で見せたい。

主たる主張: arb のデモは「相手の AI 習熟度に合わせて2モードを切り替える」
適応型である。1時間で現状の可視化→提案→実演→改善幅の確認までを完結させる。

【コンテキスト】対面1時間、相手は中小企業の経営者層、AIツール利用者と未経験者で
「響くデモ」が違うため2モードを用意

【数字・固有名詞】商談時間 1時間 / Mode A / Mode B / 100点満点スコア

【プロセス】ヒアリング → スコア診断 → arb提案 → AI更新実演 → 成果確認

【強調】「適応型2モード」のユニークさ / スコアの「改善幅」でクロージング / 1時間で完結
```

→ 結果: codex が「2モード対比をどこで見せるか、スコア可視化をどう描くか」を判断し、表 + フロー + キャラクター + コールアウトを組み合わせた情報密度の高い1枚を出す。

## スクリプト一覧

スキルの動作ロジックは `scripts/` に分散している。SKILL.md は意思決定（何をプロンプトに入れるか、どのプリセットを使うか）を担い、実行は scripts に任せる。

| スクリプト | 役割 |
|---|---|
| `scripts/generate.sh` | **メインエントリ**。これを呼ぶだけで生成完了まで一気通貫 |
| `scripts/run_codex.sh` | codex 起動ラッパー（ファイル経由 + stdin クローズで exit 144 を回避） |
| `scripts/wait_for_png.sh` | PNG 出現待ち（300秒タイムアウト + 10KB サイズ検証） |
| `scripts/notify_done.sh` | macOS 無音通知（osascript display notification） |

**外部依存**: WebP 変換は別スキル **`image-compress`** に委譲（`~/.claude/skills/image-compress/scripts/compress.sh` を直接呼び出す）。quality 95 既定で視覚的にロスレス + ファイル軽量。

各スクリプトは独立して呼べる（テスト・デバッグ用）。挙動を変えたい時は SKILL.md ではなく該当スクリプトを編集すれば全インフォグラフィックに反映される。

## ファイル名規約

**単一ページ**:
```
infographic-<topic-slug>-<YYYYMMDD-HHmm>.webp
```

**複数ページ**:
```
infographic-<topic-slug>-p1of3-<YYYYMMDD-HHmm>.webp
infographic-<topic-slug>-p2of3-<YYYYMMDD-HHmm>.webp
infographic-<topic-slug>-p3of3-<YYYYMMDD-HHmm>.webp
```

- `topic-slug`: 内容を表す短い英語（kebab-case）
- 複数ページのときは slug 引数自体に `-pNofM` を含める（例: `<topic>-p1of3`）
- 日時付きで上書きを防ぐ。同じトピックでも何回でも作り直せる
- タイムスタンプは scripts/generate.sh が内部生成するので Claude は意識不要

## 日本語テキスト崩れ対策（バランス重視）

このスキルは**必ず日本語のインフォグラフィック**を生成する。image 2.0 は日本語をそれなりに描けるが、過度にテキストを詰めると崩れる。**情報量を削って単調なテンプレ図に戻すことは避け**、以下のバランスを取る:

1. **画数の多い長い熟語を短く言い換える** — 「複雑性」→「複雑」、「設計図」→「設計」程度の言い換えはOK
2. **本文の1セクションあたり 2〜3 行を目安に** — それ以上は読みづらいし崩れやすい
3. **タイトルや見出しは堂々と日本語で書いてよい** — 短すぎる必要はない
4. **崩れた箇所があれば会話内テキストで正しい表記を補足** — 画像にキャプションは付けない
5. **生成後の確認** — Claude が Read で画像を見て、致命的な誤字・崩れ・欠字があればユーザーに知らせる

許容できないレベルで崩れた場合は、テキスト量を局所的に削る or 該当部分の言い換えで対応。**情報密度や視覚技法の多様性を犠牲にしない**。

## 失敗ハンドリング

scripts/generate.sh は明確な exit code を返す:

| exit code | 意味 | 対応 |
|---|---|---|
| 0 | 成功 | stdout の最終行を読んで報告 |
| 1 | PNG 生成タイムアウト or サイズ異常 | stdout に Codex log の tail と grep 結果が出ている → ユーザーに要約して通知し、再試行案を提示 |
| 2 | 引数エラー | slug や prompt-template ファイルの指定を確認 |
| 3 | WebP 変換失敗（cwebp/sips/pngquant 全滅）| `brew install webp` を案内 |

その他、スクリプト外で発生する事象:

- **codex コマンドが見つからない**: `npm install -g @openai/codex` を案内
- **codex がタイムアウト + log に `OPENAI_API_KEY is not set` が出てる**: プロンプトで `gpt-image-2` などモデル名を明示すると codex が CLI 経路（要 OPENAI_API_KEY）を選んでしまう。プロンプトテンプレートでは**モデル名を明示せず**「組み込みの画像生成ツール（image_gen）を使ってください」と指示する（既に修正済みのはず、再発したらここを確認）
- **codex が単調な「番号カード並列」レイアウトに逃げた**: プロンプトの「視覚化の指示」を具体化して再生成（例: 「○×の比較表を必ず使う」「タイムラインを入れる」「最上部にコールアウトを置く」など明示）
- **文字が小さくて読みにくい**: セクション数を 5〜7 に絞る、または複数枚に分割（Step 1 の分割判断ロジック参照）
- **日本語テキストが大幅に崩れた**: テキスト量を局所的に減らして再生成、または該当部分を言い換え

## やらないこと（スキルの境界）

- HTML ラッパーや Web ページ作成 → codex-design-to-code の領域
- 複数アセットの並列生成 → 1枚画像方針
- ブラウザ自動オープン → Downloads にあれば十分
- 後加工によるテキスト訂正 → 複雑化を避ける
- 設計カンプ・LP制作 → codex-design-to-code を呼ぶ
- 情報量を絞って「綺麗な空白」を作ること → 情報密度を犠牲にしない

これらに踏み込みそうになったら、いったん止めて codex-design-to-code を使うべきかユーザーに確認する。
